# LLMは「編集距離」をどこまで計算できるのか？ モーラ列・カナ列・分割の与え方を比較した

`soramimi-phonetic-search-experiments` で、LLM が **推論なし (`reasoning_effort=none`)** でどのくらいモーラ編集距離を計算できるかを試しました。

今回は特に、**入力の与え方**で精度がどれくらい変わるかを見ています。途中で気になって、

- LLM は **カタカナ文字列をモーラ列へ変換すること自体** はどれくらいできるのか
- LLM は **カナ編集距離** ならどれくらい計算できるのか
- 精度低下の原因は、**LLM の tokenizer と編集距離を取りたい単位のズレ** ではないか

も追加で調べました。

まず比較したのは次の3条件です。

1. `mora_spaced`: モーラ列をスペース区切りで明示して与える
2. `kana_only`: カタカナ文字列だけを与える
3. `kana_only_with_mora_hint`: カタカナ文字列だけを与えつつ、「ン/ー/ッは1モーラ、ジャのような拗音も1モーラ」といった簡単な説明を足す

## 結論

まず、**モーラ列をそのまま明示した場合はかなり高精度**でした。一方で、**カタカナ列だけを見せてモーラ編集距離を答えさせると精度は大きく落ちました**。

しかも今回の条件では、  
**「ン/ー/ッは1モーラ、ジャも1モーラ」のような補足説明を足しても改善しませんでした。**

追加実験まで含めると、今回の観察は次のようにまとまりました。

1. **LLM は編集距離の計算自体はかなりできる**
2. **モーラ分割も、今回の100サンプルではかなり正確にできた**
3. ただし **入力表現が曖昧なまま** だと精度が落ちる
4. **スペースで単位を明示すると大きく改善**する
5. ただし、**「tokenizer が 1 単位 1 token でないと exact match しない」という強い仮説は成り立たなかった**

## 実験設定

- モデル: `gpt-5.4`
- `reasoning_effort`: `none`
- `max_output_tokens`: `64`
- データ: `small_dataset_for_llm`
- 評価対象: **100 pair**
- サンプリング seed: **7**

同じ種類の比較では、**同じ seed=7** を使っているので、**同じサンプル**で比較しています。

正解ラベルのモーラ編集距離は `jamorasep.parse()` でモーラ列に分解したうえで、Levenshtein distance を計算して作っています。

カナ編集距離では、カタカナ文字列そのものに対して文字単位の Levenshtein distance を使っています。

## 1. モーラ編集距離: 入力の与え方でどう変わるか

| 条件 | exact_match_rate | mean_absolute_error | pearson_correlation |
| --- | ---: | ---: | ---: |
| `mora_spaced` | 0.99 | 0.01 | 0.9912 |
| `kana_only` | 0.82 | 0.18 | 0.8938 |
| `kana_only_with_mora_hint` | 0.81 | 0.19 | 0.8775 |

## 2. モーラ分割そのものはどれくらいできるか

`jamorasep` の結果を正解として、カタカナ文字列をモーラ列へ分割できるかも見ました。

| 条件 | exact_match_rate | mora_count_match_rate | mean_absolute_mora_count_error |
| --- | ---: | ---: | ---: |
| `mora_segmentation_nonreasoning` | 1.00 | 1.00 | 0.00 |

今回の100サンプルでは、**モーラ分割は全問正解**でした。

少なくともこの範囲では、

- モーラ分割が完全に苦手というわけではない
- モーラ編集距離で落ちる原因は、単に「モーラにできないから」だけではなさそう

と言えそうです。

## 3. カナ編集距離はどれくらいできるか

モーラではなく、**カタカナ文字列そのものの編集距離**も試しました。

さらに、カナを 1 文字ずつスペース区切りにした入力も比較しました。

| 条件 | exact_match_rate | mean_absolute_error |
| --- | ---: | ---: |
| `kana_only` | 0.72 | 0.30 |
| `kana_spaced` | 0.95 | 0.05 |

ここで面白いのは、**「リ ャ オ」のように 1 文字ずつスペースで区切るだけで大幅に改善した**ことです。

これは、モデルが編集距離そのものを全く計算できないというより、  
**どの単位で並べて比較するかが見えやすいと急に強くなる**ことを示唆しています。

## 4. 見えてきたこと

### 1. モーラ列を明示すると、LLMはかなり正確に距離を返せる

`mora_spaced` では、

- exact match rate: **0.99**
- MAE: **0.01**
- Pearson: **0.9912**

とかなり良い値が出ました。

つまり、**「どこでモーラが切れるか」を先に与えてしまえば、編集距離そのものの計算はかなりできる**ように見えます。

### 2. カタカナ列だけだと精度がかなり落ちる

`kana_only` では、

- exact match rate: **0.82**
- MAE: **0.18**
- Pearson: **0.8938**

でした。

この差を見ると、LLMにとって難しいのは単なる編集距離計算というより、  
**どの単位で比較するかを入力から安定して読み取る部分**なのかもしれません。

たとえば、以下のような扱いが混ざると難しさが増えます。

- 長音 `ー`
- 促音 `ッ`
- 撥音 `ン`
- 拗音 `ジャ`, `シュ`, `キョ` など

### 3. 簡単なモーラ説明を足しても今回は改善しなかった

`kana_only_with_mora_hint` では、

- exact match rate: **0.81**
- MAE: **0.19**
- Pearson: **0.8775**

で、`kana_only` より少し悪化しました。

今回は少なくとも、

- `ン / ー / ッ は1モーラ`
- `ジャ` のような拗音も1モーラ

という**短い説明を足すだけでは十分ではなかった**ようです。

これは、

- 説明が短すぎる
- 逆にプロンプトが少しノイズになった
- 100ペアというサンプルでは偶然もある

といった可能性があります。

## 5. tokenizer 仮説を検証してみた

ここまで見ていて、

> LLM は編集距離の計算自体はある程度できる。  
> ただし、LLM の tokenizer が「編集距離を求めたい単位」と一致しないと、うまく求めにくいのではないか

という仮説を持ちました。

たとえば、

- カナ編集距離なら「1文字ずつ」
- モーラ編集距離なら「1モーラずつ」

で入力が切れているほうが扱いやすいのでは、という考えです。

そこで、`tiktoken` の `o200k_base` を **proxy tokenizer** として使い、各ペアについて

- 目的単位ごとに **1 unit = 1 token** になっているか
- それが exact match とどう関係するか

を調べました。

### 結果

結論から言うと、**強い形の仮説は成り立ちませんでした**。

つまり、

> 「1単位ずつ token 化されていないペアは exact match しない」

とは言えませんでした。

実際、**strict に 1 unit = 1 token でないペアでも exact match した例がかなりあります**。

| 条件 | strict unit-token match rate | strict match群 exact rate | misaligned群 exact rate |
| --- | ---: | ---: | ---: |
| `mora_spaced` | 0.65 | 1.000 | 0.971 |
| `kana_only` | 0.64 | 0.797 | 0.583 |
| `kana_spaced` | 0.65 | 0.969 | 0.914 |

特に `mora_spaced` と `kana_spaced` では、**tokenizer 上は strict に 1 unit = 1 token でなくても高精度**でした。

### どう読むのがよさそうか

今回の結果からは、次のように読むのが自然そうです。

- **tokenizer の単位一致は精度に多少は効いていそう**
- でも、**それが exact match の必要条件ではない**
- むしろ重要なのは、**入力表現として単位境界が人間にもモデルにも見えやすいこと**

つまり、真のボトルネックは

- tokenizer が厳密に 1 unit = 1 token かどうか

というより、

- **モデルが比較単位を安定して把握できる表現になっているか**

にあるのかもしれません。

## 6. どういう解釈がよさそうか

今回の結果だけを見ると、LLMの性能はざっくり次のように整理できそうです。

- **モーラ列が与えられれば、距離計算はかなり強い**
- **カナ編集距離も、1文字ずつ区切ればかなり強い**
- **入力が曖昧なままだと精度が落ちる**
- **簡単な自然言語説明を足すだけでは、そのギャップはあまり埋まらない**
- **tokenizer の厳密一致だけでは説明しきれない**

つまり、モーラを扱うタスクでは、LLMに全部やらせるよりも、

1. まず外部ツールでモーラ列に変換する
2. その後にLLMへ渡す

という構成のほうが安定しそうです。

## 8. いまのところの感想

個人的には、

- `kana_only` のモーラ編集距離: **0.82**
- `kana_spaced` のカナ編集距離: **0.95**
- `mora_spaced` のモーラ編集距離: **0.99**

と、**単位境界をどれだけ明示するかで一気に性能が変わる**のがかなり印象的でした。

モーラや文字単位が本質的なタスクでは、プロンプト工夫だけで頑張るより、  
**前処理で比較単位を明示するほうがコスパが良い**可能性が高そうです。

## 9. 次に試したいこと

次はこんな比較をやると面白そうです。

- 別モデルでも同じ差が出るか
- few-shot を入れると `kana_only` は改善するか
- chain-of-thought や reasoning を有効にすると改善するか
- 100件ではなく、もっと大きいサンプルでも傾向が維持されるか
- tokenizer 仮説を別のモデルでも見ると同じ傾向か
- どんな誤答が多いのか（+1 / -1 のズレ、長音や拗音まわりの失敗など）
